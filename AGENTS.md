# AGENTS.md

This file provides Go-specific instructions for OpenCode when working with this repository.

## Project Overview

`betteralign` is a Go static analysis tool (fork of `fieldalignment`) that detects structs using suboptimal memory layout and optionally rewrites them. It sorts struct fields by descending alignment, then minimizes GC pointer scan overhead.

`CLAUDE.md` is a near-duplicate of this file for Claude Code; keep the two in sync when editing build or architecture notes.

## Commands

The project uses [Task](https://taskfile.dev/) as the build runner.

```sh
task test          # run all tests
task build         # regenerate, format, and build the binary
task lint          # regenerate, format, and run golangci-lint
task fmt           # format code with gci + gofumpt
task generate      # regenerate match_generated.go, then fmt
task update        # go get -u + tidy
```

Direct Go commands:
```sh
go test ./...                      # run tests
go test -run TestApply             # run a single test by name
go test ./internal/dstmin          # test only the reprint engine
go test -bench=. -benchmem ./...   # run benchmarks
go build ./cmd/betteralign         # build without ldflags
```

Linting uses `golangci-lint run --timeout 5m` with config in `.golangci.yml` (v2 format). Formatters are `gci` and `gofumpt` (not plain `gofmt`).

### Fuzzing

Native `testing.F` targets, via Task (override `FUZZTIME=2h`, set `CLEAN=true` to clear the coverage cache):

```sh
task fuzz-optimalorder           # FuzzOptimalOrder       (root: layout math)
task fuzz-gcsizes                # FuzzGCSizes            (root: size/ptrdata math)
task fuzz-decoratefileidentity   # internal/dstmin: decorate→reprint round-trips
task fuzz-decoratefilereorder    # internal/dstmin: reorder preserves comments + layout
```

The `afl-fuzz-*` tasks run the same targets under gosentry/LibAFL (`CGO_ENABLED=1`, `GOSENTRY` points at the gosentry `go` binary). `GRAMMAR=true` appends the `golang-nautilus-grammar.json` Nautilus grammar to the LibAFL flags for grammar-based generation. Plain `go test` replays the committed seed corpus under `internal/dstmin/testdata/fuzz/` as ordinary subtests, so it doubles as a regression net.

> Gotcha: `.gitignore`'s unanchored `fuzz/` pattern also ignores `testdata/fuzz/`, so committed fuzz seeds must be added with `git add -f`. Name bug seeds `bugNN_description`.

## Architecture

### Package layout

- **`betteralign` (root)** — the core `analysis.Analyzer`: alignment analysis, layout/ptrdata math, file I/O, flags.
- **`internal/dstmin/`** — the in-tree comment-preserving reprint engine, imported as `dst`. It replaced a `github.com/sirkon/dst` dependency. It decorates an `*ast.File` with per-field byte-range spans, then reprints by byte-splicing reordered struct bodies into the original source and finishing with `go/format.Source`. It supports exactly one mutation (reordering a struct's field list) and deliberately rejects shapes it cannot splice safely (single-line structs, fields sharing a line, block comments crossing a brace, mid-body build constraints / `//line` directives, bodies with `\r`/`\f`/`*//`). See `internal/dstmin/README.md`.
- **`cmd/betteralign/`** — thin CLI wrapper. Wires up resource limits (`automemlimit`, `automaxprocs`), handles `-V`/`--version`, and calls `singlechecker.Main(betteralign.Analyzer)`.

### Key files

| File | Purpose |
|------|---------|
| `betteralign.go` | Analyzer definition, `run()` entrypoint, reorder/rewrite, layout/field sort (`layoutMetrics`, `optimalOrder`, `compareFieldElem`), GC size/ptrdata (`gcSizes`) |
| `internal/dstmin/dstmin.go` | `Decorator` / `DecorateFile` / `Fprint` — decoration, comment routing, splice-based reprint |
| `match_generated.go` | **Generated** — DFA matcher for `// Code generated by … DO NOT EDIT.` comments. Regenerate with `task generate` (the `rec` tool); do not edit by hand. |
| `golang-nautilus-grammar.json` | Nautilus grammar for grammar-based LibAFL fuzzing (see Fuzzing) |
| `cmd/betteralign/main.go` | Entry point; sets GOMAXPROCS and GOMEMLIMIT from cgroup/system |

### Analysis pipeline

1. The `inspect.Analyzer` traverses the AST; `betteralign` hooks `Preorder` on `*ast.File`, `*ast.GenDecl`, and `*ast.StructType` nodes.
2. Optimality is computed from type info alone, so report-only mode never reads source bytes or builds DST. The hot path uses `layoutMetrics`, which decides optimality (and the optimal size/ptrdata) without allocating a field permutation; `optimalOrder` — which allocates the `[]int` permutation that drives the reorder — runs only under `-apply` for a misaligned struct. Both share the `compareFieldElem` comparator and a per-pass `gcSizes`, whose memoisation caches are created once per pass and are **not** concurrency-safe.
3. Only under `-apply` is a `dstmin` decorator built — lazily, on the first misaligned struct in a file. The reordered DST is serialized into a per-file buffer keyed by filename in `dirtyFiles`.
4. After the pass, if `-apply` is set, each buffer is atomically written via `github.com/google/renameio/v2/maybe` (atomic on POSIX, best-effort on Windows). `Fprint` re-parses its own gofmt output, so a successful `Fprint` already guarantees the bytes parse.

### Why a whole-file reprint instead of SuggestedFixes

Go's `ast` package does not associate comments with nodes (only byte offsets), so rewriting a single struct node loses all comments. `internal/dstmin` preserves them by reprinting the whole file. The trade-off: `SuggestedFixes` is never populated, `-fix` is a no-op alias for `-apply`, and `golangci-lint` auto-fix integration is impossible.

### Field sort order (`compareFieldElem`)

Fields are sorted **stably** by the `compareFieldElem` comparator (shared by `layoutMetrics` and `optimalOrder`):
1. Zero-sized types first
2. Higher alignment first
3. Pointer-bearing types before pointer-free types
4. Among pointer-bearing types, fewer trailing non-pointer bytes first (minimises GC ptrdata)
5. Larger size first

A multi-name field (`A, B int`) moves as one grouped declaration — the reorder never splits it.

### Composite-literal safety

A struct constructed anywhere in the package via a *positional* composite literal (`T{1, 2, 3}` rather than keyed) is reported but **never** rewritten under `-apply`, since reordering would silently re-map the literal's elements. The diagnostic points at the offending literal.

### Magic comment annotations

- `// betteralign:ignore` inside a struct's opening brace — skip this struct entirely.
- `// betteralign:check` on the `type` declaration — required when `-opt_in` is set; placing it on a `type ( … )` group opts in every spec inside.

### Testing

Analyzer tests live in `betteralign_test.go` (package `betteralign_test`) and use `golang.org/x/tools/go/analysis/analysistest`.

- `TestSuggestions` — runs the analyzer on `testdata/src/a` and checks `// want` annotations match diagnostics.
- `TestApply` — copies `testdata/src/a/*.go` to a temp dir, runs `-apply`, and compares against `*.go.golden` via `gotest.tools/v3/golden`.
- Architecture-specific files (`a_amd64.go`, `a_arm64.go`) are filtered to the current `runtime.GOARCH`.
- Flag-specific tests cover `-exclude_dirs`, `-exclude_files`, and `-opt_in` under `testdata/src/exclude/` and `testdata/src/optin/`.
- `internal/dstmin` carries its own decoration/reprint unit tests plus `TestReorderCrashersDoNotPanic`, which replays a local crash corpus.

When adding analyzer test cases: add a `.go` file under `testdata/src/a/` with `// want` annotations, and a corresponding `.go.golden` with the expected post-apply output.
